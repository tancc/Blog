Presentational and Container Component

這是一種React Component撰寫時的pattern(也稱Smart and Dumb component)，介紹可以參照Redux的文件或是Dan Abramov的文章 主要要做到的就是關注點分離。簡單來說就是 Presentational Component關心的是"how things look"，負責的就是單純view的呈現，接受父層傳下來的props，然後把view它render出來，通常為stateless。 Container Component關心的是"how things work"，負責提供data以及action的dispatch行為給子component，負責其他除了view外觀的其他邏輯，會與連接Redux的connect()類似的high order component連接，通常為stateful。

拆分成Presentational and Container Component

把關注點分開後可以更好的管理component，讓我們可以避免寫出過於擁腫且將各種邏輯混雜在一起的component，此外如此切分對開發上、測試上也有助益。 <hr /> <a name='sfc'>

Stateless Functional Component(SFC)

主要會用在Presentational Component，如上述的Presentational Component可寫作如此

直接了當的就是返回view，沒有其他東西，可以發現程式碼少了許多 除此之外在參數上的解構，可以直接明瞭的知道這個component所用的prop有哪些，整體上更增加了可讀性。

SFC本身有些限制，沒有this、沒有lifecycle methods、沒有refs，但這某種程度也是說，當component不需要這些東西的時候就很適合寫成SFC(如Presentational Component)，而將這些東西交給父層來控制。

比較大的壞處是說，因為沒有lifecycle methods，也就是沒有shouldComponentUpdate，無法用它來優化效率，因此通常會用high order component來解決，推薦搭配使用Recompose來優化。 <hr /> <a name='user_arrow_function'>

使用Arrow Function

他的好處在於代碼較簡潔(主要是省略了function字眼，許多時候也可以省略return)，而且會自動綁定this。常用的地方如

  1. action creator

  2. mapStateToProps / mapDispatchToProps

  3. map / filter

  4. SFC 上面範例可以寫成

但用在定義含有複雜邏輯的function個人會不建議使用，arrow function會是匿名函式，出事情因為在debug時會稍微麻煩一些些些，不過現在開發者工具都很好用，只是多一個步驟開他的call stack而已。 <hr /> <a name='naming_about_action'>

使用 redux-actions

1. 達到action物件內容的一致性

能讓我們所產生出的acion會滿足Flux Standard Action規範（有個工具是專門來檢查這件事的Flux-Standard-Action），達到action物件內容的一致性

  • 為單純的javascript object
  • property必須有type
  • property可能有payload、error、或meta。除此之外不會有其他property

但是使用createAction所產生的action creator會無法搭配使用redux-thunk，因為他總是會回傳action物件，這是使用時的限制。

2. 可在reducer裡引用action creator，且不用switch...case

可以不用用switch...case這種看似冗長的形式來定義reducer，可寫出更精簡的reducer。

更多用法請參照redux-actions <hr /> <a name='user_selectors'>

使用selectors

建構Redux store裡的state時，會希望是minimal possible state，也就是能表達這個app狀態的最小state結構。然後通常會在mapStateToProps來取得部分的狀態，或是根據現有狀態做運算得到所想要的狀態

但這樣做有幾個壞處：

  1. component會跟state的結構耦合在一起。若是某天state結構改變了，相對應的component他的mapStateToProps都要去修改。

  2. 每次update都會去執行mapStateToProps內的所有事情，要得到visibleList時對dataList的map，要得到costValue時所需執行的someExpensiveFunction(...)，即時此次update的對象跟visibleList、costValue之質沒有關係也一樣。

第一點也就是定義selectors所可以解決的。他可能被定義在相對應的reducer的同一隻檔案裡面，或是獨立一隻檔案跟對應的reducer放在同一層裡，而其內容很簡單，就是返回所需要的state。 通常命名法都是：get<名詞>

有時也可將所有state的selectors再放到上一層的目錄結構裡，selectors/index.js檔案

定義好selectors後mapStateToProps就可寫成

寫selectors的好處就是，這些從state獲取資料的邏輯被定義在別處，跟component解耦。就算state結構改變了，只要改寫一處selector的定義，各個地方用到相對應資料的component都不需要改變。 <hr /> <a name='user_reselect'>

使用reselect

為了解決上述第二點問題，最好的方法是使用reselect這個套件。 他能製造出對參數帶有記憶性的selectors，因此除非參數改變，否則他不做進一步的運算，直接回傳catch住的結果，所以能使效能更加提升。 此外使用reselect的selector還有可組合性，也就是可當作其他selector的input，相當方便。

使用normalizr

時常我們從後端Api取得的資料會具有巢狀結構，而巢狀結構的資料相對Reducer來說是比較不好處理的，例如需要做深層的迭代。 而normalizr就是可以幫助我們讓資料變的扁平化，利用定義Schema並根據資料的id以類似資料庫的方式，來建立對應後端回傳的資料的新的物件，方便往後的操作。

以官網的範例說明

可以看到，經過轉換後，資料變得更結構化且扁平。 result存放這筆資料所對應到的id，而完整的資料會放在entities這部分，每個entity物件也是根據他的id去存放資料。

也就是說在store中的state，若有需要存放文章資料，只需存放文章的id而不需要存放完整的文章資料。 我們可以想像，當許多sub state都存有文章完整資料的時候，我們要去更新state時，為了保持資料的一致性，需要同時更新每個用到這筆資料的sub state，十分的不方便。 而將資料結構化以後，我們可以將完整的資料內容放在一處，而其餘要用到的sub state只需存放其id或是id陣列 存取時根據id自動去mapping資料。更新時我們只要對存放完整資料的進行一次更新就好了，並且可以保證資料的一致性。
